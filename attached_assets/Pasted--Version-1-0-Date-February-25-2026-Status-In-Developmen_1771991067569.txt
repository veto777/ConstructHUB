## 1. Executive Summary

BlackHat Beacon is a web-based platform that detects, analyzes, and reports black hat SEO techniques on websites. It provides transparent website ratings (A++ to F--), evidence-backed findings with blockchain verification, and community-driven public reporting. The platform operates on a freemium model and incorporates data privacy and legal compliance features including privacy policy analysis and consent management.

---

## 2. Product Vision

### 2.1 Problem Statement
Black hat SEO practices undermine search engine integrity, harm legitimate businesses, and can expose users to security risks. Website owners, SEO professionals, and regulators lack accessible tools to identify and document these practices with verifiable evidence.

### 2.2 Solution
BlackHat Beacon provides a multi-layered detection engine that combines rule-based pattern matching, AI-powered analysis, and web crawling to identify unethical SEO techniques. Results are stored with blockchain-verified evidence chains for auditability and legal defensibility.

### 2.3 Target Users
- **SEO Professionals** — Audit client and competitor websites for compliance
- **Website Owners** — Verify their own sites are free of black hat techniques
- **Digital Marketers** — Evaluate potential partners and vendors
- **Regulators & Compliance Officers** — Investigate reports of deceptive practices
- **General Public** — Check the trustworthiness of websites before engaging

## 5. Technical Architecture

### 5.1 Technology Stack

| Layer | Technology |
|---|---|
| **Web Framework** | Flask (Python) |
| **Database** | PostgreSQL (production) / SQLite (development) |
| **ORM** | SQLAlchemy with Flask-SQLAlchemy |
| **Authentication** | Flask-Login with Werkzeug password hashing |
| **Frontend** | Jinja2 templates, Bootstrap 5, Chart.js |
| **AI/ML** | OpenAI GPT-3.5-turbo / GPT-4o (optional) |
| **Web Crawling** | Requests, BeautifulSoup4, Selenium, Scrapy |
| **Text Extraction** | Trafilatura |
| **Evidence Integrity** | Custom blockchain with SHA-256 hashing |
| **Server** | Gunicorn (WSGI) |

### 5.2 Crawler Architecture

The system uses a **Facade Pattern** with pluggable crawler strategies:

```
Crawler Facade (crawler_facade.py)
├── Unified Crawler (unified_crawler.py) — Primary, strategy-based
│   └── SimpleCrawlerStrategy — Default requests + BeautifulSoup
├── Simple Crawler (simple_crawler.py) — Fallback
├── Enhanced Crawler (enhanced_crawler.py) — Scrapy-based
└── Hybrid Crawler (hybrid_crawler.py) — Multi-strategy combination
```

**Crawler Features**:
- User-agent rotation (7 agents)
- Random request delays (0.2–1.0s) for stealth
- Error handling with metadata tracking
- Progress tracking (0–100%)
- Evidence collection with SHA-256 hashing

### 5.3 Analysis Pipeline

Detection operates across four layers, from fastest/cheapest to most sophisticated:

```
Layer 1: Rule-Based Detection (analyzer.py)
    └─ Fast, zero API cost, pattern-matching
       Detects: keyword density >5%, CSS hiding, suspicious anchors, link ratios

Layer 2: AI Analyzer Fallback (ai_analyzer.py)
    └─ Rule-based fallback when ML libraries unavailable
       Word frequency analysis, CSS pattern detection

Layer 3: SEO Content Analyzer (seo_content_analyzer.py)
    └─ OpenAI GPT-3.5-turbo/GPT-4o powered
       Full page content analysis, violation detection, rating assignment

Layer 4: Advanced AI Detector (advanced_ai_detector.py)
    └─ GPT-4o model, 16 detection types
       Catches: AI-generated spam, structured data abuse, sneaky redirects, URL manipulation
```

### 5.4 Scan Execution Flow

```
User submits URL → Website record created → ScanJob created (status: queued)
    → Background thread spawned → Crawler Facade selects crawler
    → Pages crawled recursively → Each page analyzed through detection layers
    → Evidence records created with verification hashes
    → Rating calculated from weighted severity scores
    → ScanReport stored → ScanJob marked completed
```

### 5.5 Rating System

Ratings are calculated based on the number and severity of detected issues:

| Rating | Meaning | Color |
|---|---|---|
| A++ / A+ / A / A- | Clean / Minimal issues | Green |
| B+ / B / B- | Minor concerns | Cyan |
| C+ / C / C- | Moderate issues | Yellow |
| D+ / D / D- | Significant problems | Orange |
| F+ / F / F- | Severe violations | Red |
| F-- | Extreme / Illegal activity | Dark Red (special styling) |

## 6. Data Model

### 6.1 Entity Relationship Diagram

```
User (1) ──── (N) ScanJob ──── (1) ScanReport ──── (N) Evidence
  │                  │
  │                  └──── (1) PrivacyAnalysis ──── (N) PrivacyIssue
  │
  ├──── (N) UserConsent
  └──── (N) AuditLog

Website (1) ──── (N) ScanJob

BlackHatTechnique (reference table)
```

### 6.2 Core Tables

| Table | Purpose | Key Fields |
|---|---|---|
| **user** | Authentication & profiles | username, email, password_hash, created_at |
| **website** | Scanned domains | url, domain, first_scanned, last_scanned, latest_rating |
| **scan_job** | Scan execution tracking | scan_id, user_id, website_id, status, rating, score, progress |
| **scan_report** | Detailed scan results | scan_job_id, raw_data (JSON), summary, issues_count |
| **evidence** | Individual findings | issue_type, location, description, confidence, severity, verification_hash |
| **blackhat_technique** | Technique reference DB | name, description, severity, category, weight, ethical_score, legal_score, is_illegal |
| **user_consent** | GDPR consent tracking | user_id, consent_type, consented, ip_address |
| **audit_log** | Action audit trail | user_id, action_type, action_details, ip_address, user_agent |
| **privacy_analysis** | Privacy scan results | has_privacy_policy, gdpr_compliance_score, ccpa_compliance_score, tracking_methods_detected |
| **privacy_issue** | Privacy findings | issue_type, description, severity, recommendation, references |

### 6.3 Blockchain Evidence Storage

File-based immutable evidence chain (`blockchain_data/evidence_chain.json`):
- Genesis block + sequential evidence blocks
- Each block contains: index, timestamp, transactions (evidence), previous_hash, hash, nonce
- Hyperledger Fabric-inspired endorsements from simulated nodes
- SHA-256 chain integrity verification

|---|---|---|
| Keyword Stuffing | Density analysis (>5% threshold), repetitive sequences | Medium–High |
| Hidden Text/Links | CSS property detection (display:none, visibility:hidden, font-size:0) | High |
| Thin/Duplicate Content | Content-to-code ratio analysis | Medium |
| AI-Generated Spam | GPT-4o pattern analysis | High |

| Technique | Detection Method | Severity |
|---|---|---|
| Cloaking | User-agent differentiation, JavaScript-based content swaps | Critical |
| Doorway Pages | Thin content + excessive links ratio | High |
| Sneaky Redirects | Redirect chain analysis | High |

### 9.3 Link Manipulation
| Technique | Detection Method | Severity |
|---|---|---|
| Link Farming | External link density, domain repetition | High |

### 9.4 Technical Abuse
| Technique | Detection Method | Severity |
|---|---|---|

## 12. Non-Functional Requirements

### 12.1 Performance
- Scan progress updates in real-time (0–100%)
- Quick scan results within 10–30 seconds
- Full scans support up to 500 pages
- Request delays (0.2–1.0s) for responsible crawling

### 12.2 Reliability
- Automatic crawler fallback (enhanced → simple → unified)
- Graceful error handling with metadata tracking
- Database connection pooling with pre-ping health checks
- Pool recycling every 300 seconds

### 12.3 Scalability
- Modular crawler architecture supports adding new strategies
- Analysis pipeline layers can be enabled/disabled independently
- Background thread scanning for non-blocking operation

### 12.4 Cost Optimization
- Rule-based analysis operates at zero API cost
- AI features are optional and degrade gracefully
- No paid API required for core functionality
- OpenAI integration is supplementary, not mandatory
